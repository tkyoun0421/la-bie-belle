# AI 하네스 스펙

생성된 이슈 하나를 구현 가능한 상세 스펙으로 구체화한다.

이 스킬은 `$deep-interview`를 활용한 "질문을 통한 스펙 확정"이 핵심이다. 구현 정책, 폴더 구조, 네이밍, 테스트 범위, CI 범위처럼 되돌리기 어려운 선택을 Codex가 단독으로 정하지 않는다.

## 선행 조건

다음 파일이 없으면 진행하지 말고 누락 파일을 보고한다.

- `.agents/runs/issue-{number}/task-spec.md`
- `.agents/runs/issue-{number}/plan.md`

## 절차

1. 원본 GitHub Issue, `task-spec.md`, `plan.md`를 읽는다.
2. GitHub Issue 본문과 코멘트, `task-spec.md`, `plan.md`에서 구현 전 확정해야 할 결정을 뽑아 "질문 큐"로 정리한다.
3. 질문 큐에 미확정 항목이 있으면 `spec.md`를 작성하거나 갱신하지 말고 `$deep-interview` 스킬을 함께 사용한다.
4. `$deep-interview`의 형식에 따라 `현재 이해`, `막힌 결정`, `추천 답안`, `질문`을 제시한다.
5. 질문은 한 번에 한 주제만 다룬다.
6. 각 질문에는 왜 결정이 필요한지, 선택지가 구현에 어떤 영향을 주는지 짧게 설명한다.
7. 사용자가 답하면 그 답을 현재 스펙 결정으로 요약하고, 다음 미확정 질문으로 넘어간다.
8. 모든 필수 결정이 확정되면 정상 흐름과 엣지케이스를 구분한다.
9. 에러, 빈 상태, 로딩 상태, 권한, 입력 검증, 데이터 규칙을 필요한 만큼 정의한다.
10. 테스트 케이스를 작성하고 Red 단계에서 먼저 작성할 실패 테스트 우선순위를 정한다.
11. 확정된 내용만 반영해 `spec.md`만 작성하거나 갱신한다.

## 질문 규칙

- 사용자가 명시적으로 "네가 정해", "합리적으로 기본값으로 진행해", "추천안으로 작성해"라고 하지 않으면 중요한 결정을 임의로 확정하지 않는다.
- 선택지가 분명하면 추천안을 하나 제시할 수 있지만, 추천안은 결정이 아니다. 사용자 확인을 받은 뒤 확정한다.
- GitHub Issue 코멘트에 `다음 작업 전 결정할 것`, `결정 필요`, `구현 전에`, `스펙 확정`, `폴더 구조`, `네이밍 컨벤션`, `PRD`, `커밋 메시지`, `commitlint`, `CI 범위`가 있으면 반드시 질문 큐에 포함한다.
- 이미 사용자가 답한 항목은 다시 묻지 말고, 답변 요약만 확인한다.
- 사용자의 답이 모호하면 바로 `spec.md`를 쓰지 말고 좁은 후속 질문을 한다.
- 질문은 `$deep-interview`의 원칙대로 가장 큰 불확실성 하나부터 묻는다.
- 질문은 진행을 돕는 순서로 한다: 범위와 우선순위, 구조, 네이밍, 기술 선택, 테스트/검증, DB, 커밋/CI, 문서화.
- 질문 중에는 코드, 테스트, 검증, 리뷰, 대시보드 작업을 하지 않는다.

## 미확정 항목 보고 형식

미확정 항목이 있으면 다음 형식으로 짧게 보고하고 첫 질문을 한다.

```md
스펙 확정 전 질문이 필요합니다.

확인한 미확정 항목:
- ...

먼저 결정할 항목: ...
질문: ...
```

필요하면 선택지를 제시한다.

```md
추천 선택지:
- A: ...
- B: ...
- C: ...
```

## 산출물

- `.agents/runs/issue-{number}/spec.md`

## 금지

- 코드를 수정하지 않는다.
- 테스트를 작성하지 않는다.
- Green 구현을 시작하지 않는다.
- 검증이나 리뷰를 수행하지 않는다.
- 필수 결정이 미확정인 상태에서 `spec.md`를 작성하거나 갱신하지 않는다.
