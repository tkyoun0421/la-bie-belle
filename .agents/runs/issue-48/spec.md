# 상세 스펙

## 확정 결정

- 기존 `review-score.json` 필수 필드는 유지한다.
- 카테고리별 하위 점수 항목은 `subcriteria[]` 선택 필드로 추가한다.
- `subcriteria[]`가 있는 카테고리는 상위 `score`와 `max_score`가 하위 항목 합계로 계산되어야 한다.
- 각 하위 항목은 내부 품질 기준 `quality_levels.full`, `partial`, `minimal`, `zero`를 가져야 한다.

## 세부 배점 구조

### 요구사항 충족도 25점

- 명시 요구사항 반영: 10점
- 완료 기준 충족: 7점
- 사용자 의도와 우선순위 반영: 5점
- 누락/회귀 없음: 3점

### 범위 준수 15점

- 요청 범위 안에서 해결: 7점
- 불필요한 변경 최소화: 4점
- 제외 범위 준수: 4점

### 구현 품질 15점

- 기존 패턴과 구조 활용: 5점
- 단순하고 유지보수 가능한 코드: 5점
- 오류 처리와 경계 조건 고려: 3점
- 중복/임시방편 최소화: 2점

### 검증 충분성 10점

- 자동 검증 실행: 4점
- 요구사항별 검증 연결: 3점
- 실패/잔여 리스크 기록: 2점
- 수동 확인 필요성 판단: 1점

### 리스크와 안전성 5점

- 데이터/상태/권한 위험 회피: 2점
- 기존 사용자 흐름 보호: 2점
- 파괴적 작업 회피: 1점

### 요구사항 해석 품질 10점

- 이슈 본문과 댓글 반영: 4점
- 미확정 결정 식별: 3점
- 확정된 결정만 구현에 반영: 3점

### 계획 적절성 8점

- 작업 순서 적절성: 3점
- 검증 전략 포함: 3점
- 리스크와 제외 범위 명시: 2점

### 컨텍스트 활용 6점

- 관련 파일과 기존 패턴 확인: 3점
- 이전 run/하네스 규칙 반영: 2점
- 최신 원격/이슈 상태 확인: 1점

### 기록과 인수인계 품질 6점

- 산출물 완성도: 2점
- 변경 이유와 결과 기록: 2점
- 다음 작업/감점 사유 연결: 2점

## 데이터 규칙

- `categories[].subcriteria`는 선택 항목이다.
- `subcriteria[]`가 있으면 모든 항목은 `id`, `label`, `score`, `max_score`, `evidence`, `deductions`를 가진다.
- 각 `subcriteria[]` 항목은 점수 판단 기준을 고정하기 위해 `quality_levels`를 가진다.
- `subcriteria[].max_score` 합계는 `category.max_score`와 같아야 한다.
- `subcriteria[].score` 합계는 `category.score`와 같아야 한다.
- `deductions[]`는 기존 문자열 또는 객체를 허용한다.
- 객체형 감점은 `reason` 필수, `points`, `follow_up`, `severity` 선택 필드를 가진다.
- 객체형 감점의 `points`가 있으면 0보다 큰 숫자여야 한다.
