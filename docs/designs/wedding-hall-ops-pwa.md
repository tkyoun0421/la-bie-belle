## Problem Statement
웨딩홀 팀장은 카톡, 엑셀, 전화로 근무 공고, 신청, 취소, 대타, 출근, 급여를 관리한다. 상태가 구조화된 데이터로 남지 않아서 취소가 발생했을 때 가능한 사람을 바로 찾기 어렵고, 포지션 재배치와 예상 급여 계산도 매번 수작업으로 다시 맞춰야 한다. 팀원도 자신의 스케줄, 출근 기록, 예상 급여를 한 화면에서 확인하지 못한다.

## What Makes This Cool
이 제품의 핵심은 스케줄 표가 아니라 운영 루프 자동화다. 팀장이 근무 공고를 열고 팀원들이 신청하며, 취소가 생기면 가능한 인원에게 즉시 알림이 가고, 선착순 또는 승인 방식으로 다시 채워진다. 여기에 체크인과 예상 급여가 붙으면 팀장은 운영 효율을 느끼고, 팀원은 앱을 계속 써야 할 이유를 느낀다.

## Constraints
- 이 프로젝트는 스타트업 검증보다 사이드 프로젝트 성격이 강하다.
- 첫 버전부터 팀장과 팀원 모두가 효용을 느껴야 한다.
- 메인 데모는 스케줄 관리 흐름이지만, 위치 체크인은 팀원 채택을 만드는 핵심 기능이다.
- 첫 버전의 위치 체크인은 백그라운드 자동 감지가 아니라, 사용자가 앱에서 직접 실행하는 수동 위치 인증 체크인으로 시작한다.
- 첫 버전은 체크인만 받고 체크아웃은 받지 않는다. 체크인의 목적은 출근 증빙이다.
- 알림 채널은 첫 버전부터 웹 푸시만 사용한다. 카카오톡이나 SMS는 백업 채널로 두지 않는다.
- 첫 버전의 권한 모델은 `어드민 / 팀장 / 팀원` 3역할로 고정한다. 어드민은 슈퍼 어드민 성격으로 시스템 설정과 조직 관리를 맡고, 현장 운영은 팀장이 담당한다.
- 첫 버전의 급여 규칙은 `시급 x 인정 근무시간`을 기본으로 한다. 인정 근무시간은 `첫 식 시작 - 2시간`부터 `막식 종료 + 2시간`까지로 계산하고, 한 근무가 9시간을 초과하면 초과분은 1.5배로 계산한다.
- 구현 스택은 `Next.js + Google Auth 로그인`으로 고정한다.
- 첫 버전은 단일 팀인 `라비에벨 예도팀`만 대상으로 한다.
- 앱스토어 출시보다 한 팀에 바로 배포할 수 있는 형태가 우선이다.
- 주말 단위로 만들고 실제 팀에서 바로 돌려볼 수 있어야 한다.

## Premises
1. 이 제품의 핵심 문제는 근무표 작성 자체보다 취소와 결원을 실시간으로 복구하는 운영이다.
2. 첫 배포는 앱스토어보다 URL로 바로 배포 가능한 웹앱/PWA가 맞다.
3. 급여는 첫 버전부터 정식 급여 시스템이 아니라 예상 급여 계산과 내보내기로 시작하는 게 맞다.
4. 메인 데모는 스케줄 관리 흐름이지만, 위치 체크인은 팀원 입장에서 앱을 계속 쓸 이유를 만드는 채택 기능이다.
5. 첫 버전의 위치 체크인은 자동 추적이 아니라, 사용자가 현장에서 `출근하기`를 눌렀을 때 현재 위치를 검증하는 방식으로 한다.
6. 첫 버전은 체크인만 받고 체크아웃은 받지 않는다. 체크인의 역할은 급여 계산이 아니라 출근 확인이다.
7. 취소와 대타 모집 알림은 웹 푸시만으로 처리한다. 따라서 홈 화면 설치와 알림 권한 허용은 온보딩의 필수 단계다.
8. 권한 모델은 `어드민 / 팀장 / 팀원`으로 나눈다. 어드민은 전체 시스템과 조직 설정을 관리하고, 팀장은 실제 행사 운영을 담당한다.
9. 첫 버전의 급여는 `시급 x 인정 근무시간`으로 계산한다. 인정 근무시간은 `첫 식 시작 - 2시간`부터 `막식 종료 + 2시간`까지로 잡고, 9시간 초과분은 1.5배로 계산한다.
10. 팀장은 예외 케이스에서만 급여 계산 결과를 수동 수정할 수 있다. 수동 수정은 기본 흐름이 아니라 드문 운영 예외 처리 수단이다.
11. 행사 시간 모델은 `첫 식 시작 시각`과 `막식 종료 시각` 두 값으로 저장한다. 급여 규칙과 운영 계산은 이 두 값을 기준으로 한다.
12. 행사 시간은 팀장이 매번 완전 수동 입력하는 대신, 자주 쓰는 행사 템플릿을 선택한 뒤 필요하면 수정하는 방식으로 입력한다.
13. 취소와 대타 모집 알림은 전체 브로드캐스트가 아니라, 해당 포지션을 수행할 수 있는 인원에게만 보낸다.
14. 대타는 자동 선착순 확정이 아니라, 선착순 신청 후 팀장이 최종 승인하는 방식으로 확정한다.
15. 로그인 인증은 Google Auth로 처리한다.
16. 누구나 Google 로그인은 가능하지만, 첫 로그인 후 어드민 승인을 받아야 팀 멤버로 연결된다.
17. 첫 로그인 승인은 조직 단위로 처리하되, 첫 버전에서는 어드민 승인 시 `라비에벨 예도팀`에 바로 배정한다.
18. 첫 버전의 팀 범위는 `라비에벨 예도팀` 하나로 고정한다. 멀티팀 운영은 범위 밖이다.
19. 첫 버전 체크인 반경 기본값은 행사장 기준 150m로 시작한다. 위치 정확도가 너무 낮으면 재시도를 유도하고, 계속 실패하면 팀장 예외 승인으로 처리한다.
20. 첫 버전은 단일 홀에서 예식만 운영하는 전제를 두고, 행사 템플릿은 `예식 시간 라벨 + 첫 식 시작/막식 종료 + 기본 포지션 슬롯` 조합으로 저장한다.
21. 포지션 교육 슬롯은 템플릿 기본값에 넣지 않고, 교육이 필요한 행사에서만 팀장이 임시로 추가한다.
22. 급여 수정과 체크인 예외 승인은 항상 `수정 전 값 / 수정 후 값 / 사유 / 수정자 / 수정 시각`을 감사 로그로 남긴다.
23. 체크인 실패 시 팀원은 즉시 예외 요청을 보낼 수 있고, 팀장이 승인하면 출근 인정으로 처리한다. 행사 시작 후 30분이 지나도 체크인이 없으면 `미체크인 경고` 상태로 올린다.
24. 팀원은 홈 화면 설치와 웹 푸시 허용을 완료하기 전까지 신청, 취소, 체크인 같은 핵심 기능에 접근할 수 없다.
25. 대타 모집은 자동 마감 시간을 두지 않고, 팀장이 필요할 때 수동으로 닫는다.
26. 어드민 콘솔과 실제 운영/근무 화면은 분리한다. 승인, 설정, 포지션 자격, 템플릿, 급여 규칙 같은 저빈도 고권한 작업은 별도 어드민 영역에서만 다룬다.
27. 로그인 후 첫 진입 화면은 `공통 메인 대시보드`로 통일한다. 어드민, 팀장, 팀원 모두 같은 홈 구조를 보되, 역할에 따라 강조 카드와 빠른 액션만 달라진다.

## Approaches Considered
### Approach A: PWA 올인형
한 개의 PWA 안에서 팀장과 팀원이 모두 공고 생성, 신청, 취소, 포지션 배정, 확정, 체크인, 예상 급여 확인까지 수행한다. 가장 빠르게 배포 가능하고 팀 전체가 같은 링크로 시작할 수 있다.

장점:
- 이번 주말 데모에 가장 잘 맞는다.
- 앱스토어 없이 바로 배포할 수 있다.
- 팀장과 팀원 경험이 하나의 흐름으로 이어진다.

단점:
- 웹 푸시, 위치 권한, 백그라운드 동작에 한계가 있다.
- 기능이 늘수록 화면 설계가 금방 복잡해진다.

### Approach B: 운영 엔진 우선형
핵심은 결원 복구 엔진이다. 공고, 신청, 취소, 대타 후보 계산, 선착순 확정, 포지션 재배치를 강하게 만들고 체크인과 급여는 얇게 붙인다.

장점:
- 운영 자동화라는 10x 방향에 가장 충실하다.
- 팀장이 느끼는 효율이 가장 크다.
- 규칙 기반 자동 추천으로 확장하기 쉽다.

단점:
- 팀원 입장에서는 초반 효용이 약하게 느껴질 수 있다.
- 채택을 만드는 체크인, 급여, 공정성 경험이 약해질 수 있다.

### Approach C: 채택 우선형 운영 시스템
스케줄 신청, 취소, 대타 확정 흐름을 유지하되, 수동 위치 인증 체크인과 예상 급여 반영을 전면에 둔다. 팀장은 운영 자동화를 얻고, 팀원은 출근 기록과 급여 가시성 때문에 매일 앱을 열게 된다.

장점:
- 팀장과 팀원 모두가 첫 버전에서 효용을 느낀다.
- 공정성, 출근 기록, 예상 급여가 팀원 채택을 만든다.
- 스케줄 앱이 아니라 실제 운영 시스템처럼 느껴진다.

단점:
- 위치 권한, 체크인 신뢰성, 오탐 처리 등 구현 난도가 올라간다.
- iPhone PWA 특성상 홈 화면 설치와 권한 허용 UX를 잘 설계하지 않으면 알림과 체크인 채택률이 떨어질 수 있다.
- 제품 메시지가 분산되면 메인 가치를 흐릴 수 있다.

## Recommended Approach
Approach C를 추천한다. 이유는 이 프로젝트의 성공 조건이 단순한 관리자 도구가 아니라 팀 전체 채택이기 때문이다. 팀장은 공고, 신청, 취소, 대타 확정 자동화에서 가치를 느끼고, 팀원은 체크인과 예상 급여 때문에 앱을 계속 쓰게 된다. 즉 운영 자동화와 일일 효용이 동시에 살아야 한다.

첫 데모의 중심 흐름은 다음과 같다.
1. 팀장이 행사 공고를 만들고 필요한 포지션과 인원을 연다.
2. 팀원들이 앱에서 신청한다.
3. 팀장이 포지션을 배분하고 확정 알림을 보낸다.
4. 취소가 발생하면 가능한 인원에게 즉시 알림이 간다.
5. 같은 포지션 가능 인원이 선착순으로 신청하고, 팀장이 그중 한 명을 최종 승인해 대타가 확정된다.
6. 현장에서는 팀원이 앱에서 `출근하기`를 눌렀을 때 현재 위치를 검증하는 수동 위치 인증 체크인으로 출근이 기록된다.
7. 개인 화면에 예상 급여가 바로 반영된다.
8. 포지션별로 정규 인원과 교육 인원을 분리해서 열 수 있고, 교육이 필요한 날에는 교육 슬롯을 함께 운영할 수 있다.

권한별 책임은 다음과 같다.
- 어드민: 조직 생성, 팀장 계정 관리, 행사장 위치 반경 설정, 급여 규칙 설정, 전체 데이터 조회
- 팀장: 공고 생성, 신청 승인/배정, 취소 대응, 대타 확정, 체크인 예외 처리, 팀 단위 급여 확인, 예외 급여 수동 수정
- 팀원: 신청, 취소, 체크인, 개인 스케줄 확인, 개인 예상 급여 확인

## Success Criteria
- 팀장이 실제 행사 하나를 앱에서 공고 생성하고 포지션 배정까지 끝낼 수 있다.
- 팀장이 행사 생성 시 `첫 식 시작`과 `막식 종료`를 입력하거나 선택해서 근무시간과 예상 급여를 자동 계산할 수 있다.
- 팀장이 자주 쓰는 행사 템플릿을 선택해서 행사 생성 시간을 빠르게 채우고, 필요하면 현장 상황에 맞게 수정할 수 있다.
- 행사 템플릿은 `예식 시간 라벨 + 시간대 + 기본 포지션 슬롯` 기준으로 재사용되고, 단일 홀 운영에서는 별도 홀 축 없이도 충분히 운영 가능해야 한다.
- 팀장이 포지션별 정규 인원과 교육 인원을 따로 열고, 교육이 필요한 팀원을 함께 배정할 수 있다.
- 신규 사용자가 Google 로그인 후 승인 대기 상태를 거쳐 어드민 승인 뒤 팀원 기능에 접근할 수 있다.
- 신규 사용자가 Google 로그인 후 승인 대기 상태를 거쳐 어드민 승인과 동시에 `라비에벨 예도팀`에 배정된 뒤 기능에 접근할 수 있다.
- 팀원은 홈 화면 설치와 웹 푸시 허용을 마치기 전까지 신청/체크인 기능이 잠겨 있고, 온보딩 완료 후에만 핵심 기능을 쓸 수 있다.
- 팀원이 자신의 신청 상태, 확정 상태, 체크인 상태, 예상 급여를 한 화면에서 볼 수 있다.
- 어드민은 별도 콘솔에서 승인 대기, 팀 기본 설정, 포지션 자격, 템플릿, 급여 규칙을 관리하고, 이 기능들이 팀장 홈이나 팀원 홈에 섞이지 않는다.
- 어드민 콘솔의 기본 섹션은 `승인 대기`, `팀원-포지션 자격`, `예식 템플릿`, `급여 규칙` 네 가지로 시작한다.
- 공통 메인 대시보드에는 `오늘 예식`, `내 근무/내 상태`, `공지`, `빠른 액션`이 공통으로 들어가고, 역할별로 노출 우선순위만 달라진다.
- 취소 발생 시 가능한 인원에게 알림이 가고, 대타 확정이 앱 안에서 완료된다.
- 취소 알림은 해당 포지션 가능 인원에게만 가고, 불필요한 전체 알림 스팸이 줄어든다.
- 대타는 선착순 신청 후 팀장이 최종 승인해서 확정된다.
- 대타 모집은 자동 마감 없이 열려 있고, 팀장이 필요할 때 수동으로 닫아 운영할 수 있다.
- 체크인 기록은 타임스탬프와 위치 검증 결과를 남기고 출근 확인 근거로 사용된다.
- 체크인은 행사장 기준 150m 반경 안에서 통과되고, GPS 품질이 나쁘면 재시도 또는 팀장 예외 승인으로 복구할 수 있다.
- 체크인 실패 시 팀원은 예외 요청을 보낼 수 있고, 행사 시작 후 30분이 지나도 체크인이 없으면 팀장 화면에서 `미체크인 경고` 상태로 보여야 한다.
- 체크아웃 없이도 팀 운영과 급여 계산이 가능해야 한다.
- 예상 급여는 `첫 식 시작 - 2시간`부터 `막식 종료 + 2시간`까지의 인정 근무시간을 기준으로 계산되고, 9시간 초과분은 1.5배 규칙으로 일관되게 반영된다.
- 팀장 수동 수정과 체크인 예외 승인은 모두 감사 로그에 남아서 누가 왜 바꿨는지 추적할 수 있어야 한다.
- 팀원 전원이 홈 화면 설치와 알림 권한 허용을 마쳐서, 취소 알림이 웹 푸시만으로도 운영 가능한 수준으로 도달한다.
- 실제 한 팀이 최소 한 주 이상 카톡과 엑셀을 줄이고 이 앱을 병행 사용해 볼 수 있다.

## Distribution Plan
첫 배포는 비공개 PWA로 한다. 팀장과 팀원에게 URL을 공유하고 홈 화면 설치를 유도한다. 앱스토어 심사와 네이티브 배포는 나중 문제다.

권장 배포 형태:
- 프론트엔드: Next.js 기반 PWA
- 백엔드: 실시간 상태 변경과 인증이 쉬운 BaaS 계열
- 배포: main 브랜치 머지 시 자동 웹 배포

초기 운영 방식:
- 팀 카카오톡방에 링크 배포
- Google 로그인 후 역할별 화면 제공
- 첫 로그인 사용자는 승인 대기 화면으로 보내고, 어드민 승인과 동시에 `라비에벨 예도팀` 멤버십을 활성화
- 첫 버전의 대상 팀은 `라비에벨 예도팀` 하나
- 홈 화면 설치와 알림 권한 허용을 온보딩에서 완료해야 신청/체크인 등 핵심 기능으로 넘어가게 설계
- 취소, 대타 모집, 확정 알림은 웹 푸시만 사용

## Next Steps
1. 도메인 모델부터 고정한다. 행사, 공고, 포지션, 신청, 확정, 취소, 체크인, 예상 급여 항목을 정의한다.
   그리고 `어드민 / 팀장 / 팀원` 역할별 읽기/쓰기 권한을 표로 정리한다.
   행사 모델에는 `첫 식 시작 시각`과 `막식 종료 시각`을 필수 필드로 둔다.
   그리고 행사 템플릿 모델을 추가해 `예식 시간 라벨 + 첫 식 시작 / 막식 종료 + 기본 포지션 슬롯` 조합을 재사용 가능하게 만든다.
   팀원-포지션 관계에는 `가능 / 교육중` 상태를 두고, 행사 포지션 슬롯에는 교육 인원 수를 별도로 둔다.
   Google 로그인 후 승인 대기와 승인 완료 상태를 표현하는 멤버십 흐름도 같이 정의한다.
   어드민 승인과 동시에 `라비에벨 예도팀`에 배정되는 흐름으로 정리한다.
   팀원은 홈 화면 설치와 웹 푸시 허용을 끝내야만 신청/체크인 기능으로 진입할 수 있게 온보딩 게이트를 둔다.
   그리고 어드민 콘솔과 실제 운영/근무 앱을 라우트와 네비게이션 수준에서 분리한다.
   어드민 콘솔은 `승인 대기`, `팀원-포지션 자격`, `예식 템플릿`, `급여 규칙` 네 섹션을 기준으로 화면을 구성한다.
   공통 메인 대시보드는 어드민/팀장/팀원이 함께 쓰는 첫 화면으로 설계하고, 역할별 세부 작업 화면은 그 다음 레벨로 둔다.
2. 첫 데모 3화면을 설계한다. 팀장 공고/배정 화면, 팀원 신청/취소/예상 급여 화면, 취소 후 대타 모집/확정 화면.
3. 취소 발생 시 가능한 인원에게 알림을 보내는 상태 전이 규칙을 정의한다.
   첫 버전은 전체 브로드캐스트가 아니라, 동일 포지션 가능 인원 필터만 적용한다.
   대타는 자동 확정하지 않고, 선착순 신청 후 팀장이 승인하는 흐름으로 고정한다.
4. 수동 위치 인증 체크인 규칙을 정한다. 허용 반경, 권한 거부 시 예외 처리, 실내 오차 대응, 실패 시 재시도 UX를 정의한다.
   첫 버전은 `150m 반경`을 기본값으로 두고, GPS 정확도가 너무 낮으면 재시도 후 팀장 예외 승인으로 마무리한다.
   체크인 실패 시 팀원이 바로 예외 요청을 보낼 수 있고, 행사 시작 후 30분이 지나도 체크인이 없으면 팀장 화면에 미체크인 경고를 띄운다.
5. 예상 급여 계산기를 붙인다. `첫 식 시작 - 2시간`, `막식 종료 + 2시간`, 9시간 초과분 1.5배 규칙으로 자동 계산하고, 팀장이 예외 케이스를 수동 수정할 수 있게 한다.
6. 체크인 실패, 위치 권한 거부, 미체크인 상태에 대한 팀장 예외 승인 흐름을 구현한다.
7. 실제 팀 한 곳에서 파일럿을 돌려서 카톡과 엑셀을 얼마나 대체하는지 본다.

## What I noticed about how you think
- 너는 초반부터 문제를 추상적으로 보지 않았다. `웨딩홀 근무 스케쥴 관리 애플리케이션`이라고 시작했지만, 금방 `카톡 엑셀 전화지`라는 실제 운영 도구 조합으로 내려왔다.
- `내가 팀장이야`라는 말이 이 설계의 기준점을 잡았다. 관리자 화면만 멋지게 만드는 쪽으로 안 가고, 실제 운영자가 무엇을 제일 귀찮아하는지 계속 붙잡았다.
- `모든 팀원들이 이 사이드 프로젝트에 효율성을 느껴야해`라는 문장이 이 제품의 방향을 바꿨다. 관리자 도구가 아니라 팀 전체 채택이 필요한 제품이라는 점이 선명해졌다.
- `위치 체크인이 있어야 팀원들이 이 앱을 사용하는데 효용감을 느낄 거 아냐`라는 말에서, 너는 기능 수보다 반복 사용 이유를 더 중요하게 본다. 그 감각은 맞다.
