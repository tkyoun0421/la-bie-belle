# 라비벨 제품 PRD

## 제품 요약

라비벨은 하나의 웨딩홀에서 예도팀 근무를 관리하는 PWA 애플리케이션이다. 웨딩홀 관리자, 예도팀장, 예도팀원이 같은 근무일 데이터를 기준으로 모집, 신청, 배정, 확정, 확인, 완료 흐름을 처리한다.

## 사용자 역할

- 웨딩홀 관리자: 슈퍼 어드민이다. 팀장 지정, 팀원 관리, 근무일 관리, 배정, 확정, 완료 상태 확인을 할 수 있다.
- 예도팀장: 근무 운영자다. 팀원 정보 관리, 신청자 확인, 포지션 배정, 근무 확정, 근무 완료 관리를 할 수 있다.
- 예도팀원: 모집 중인 근무일에 신청하고, 확정된 근무와 배정 포지션을 확인한다.

## MVP 1 범위

- 하나의 웨딩홀 조직만 관리한다.
- 관리자가 초대 링크를 보내고, 초대받은 사용자는 구글 소셜 로그인으로 가입한다.
- 관리자는 팀장을 지정할 수 있다.
- 근무는 하루 단위로 생성한다.
- 모집 중인 근무일에 팀원이 직접 신청한다.
- 모집 중에는 팀원에게 날짜, 요일, 모집 상태, 내 신청 여부만 보여준다.
- 포지션 배정 단계에서 그날의 예식 시간들을 선택한다.
- 선택된 예식 시간은 출퇴근 시간 자동 계산, 근무 규모 기록, 팀원에게 식 개수와 시간 정보를 알리는 데 사용한다.
- 포지션 배정은 예식 시간별이 아니라 하루 전체 포지션 기준으로 한다.
- 팀장 또는 관리자가 팀원을 픽하면 근무는 즉시 확정된다.
- 팀원이 확정된 스케줄 상세를 열어보면 자동으로 확인 처리된다.
- 근무 시간이 지나면 근무는 자동으로 완료 상태가 된다.
- 출근 시간과 퇴근 시간은 자동 계산되지만, 현장 변수에 대비해 수동 수정할 수 있다.
- 앱 내부 알림과 PWA 푸시 알림은 MVP 필수 범위다.

## MVP 1 제외 범위

- 급여 정산, 지급 상태, 역할별 단가, 예외 수당 계산은 제외한다.
- 팀원 간 근무 변경 요청, 대타 조율, 팀장 컨펌 흐름은 후속 기능으로 둔다.
- 카카오톡 알림, SMS 알림은 제외한다.
- 여러 웨딩홀, 여러 지점, 여러 홀 관리는 제외한다.
- 여러 포지션 템플릿 저장, 복제, 버전 관리는 제외한다.
- 팀원이 본인의 성별이나 가능 포지션을 직접 수정하는 기능은 제외한다.

## 핵심 흐름

1. 관리자가 팀원을 초대한다.
2. 초대받은 사용자가 구글 소셜 로그인으로 가입한다.
3. 관리자 또는 팀장이 근무일을 생성하고 모집 상태로 둔다.
4. 팀원이 모집 중인 근무일에 신청한다.
5. 관리자 또는 팀장이 배정 화면에서 신청자 목록을 기본으로 본다.
6. 필요한 경우 전체 팀원 보기로 전환해 신청하지 않은 팀원도 직접 배정한다.
7. 관리자 또는 팀장이 그날의 예식 시간들을 선택한다.
8. 시스템이 출근 시간과 퇴근 시간을 자동 계산한다.
9. 관리자 또는 팀장이 필요하면 출퇴근 시간을 수동 수정한다.
10. 관리자 또는 팀장이 하루 전체 포지션 기준으로 팀원을 배정한다.
11. 배정 즉시 근무는 확정되고 팀원에게 앱 내부 알림과 PWA 푸시 알림이 발송된다.
12. 팀원이 확정 스케줄 상세를 열면 자동으로 확인 처리된다.
13. 근무 시간이 지나면 자동으로 근무 완료 상태가 된다.

## 포지션 기본값

MVP 1에서는 기본 포지션 템플릿을 불러온 뒤, 해당 근무일에서만 포지션과 인원을 조정한다. 기본 템플릿 자체 편집과 여러 템플릿 저장은 후속 기능이다.

- 팀장: 1명, 남/여
- 스캔: 1명, 남/여
- 메인: 1명, 남/여
- 드레스: 1명, 여
- 드레스실: 1명, 여
- 축가: 1명, 성별 추후 확정
- 대기실: 1명, 여
- 매니저: 2명, 남/여
- 안내: 2명, 남

각 포지션에는 교육 인원을 추가할 수 있다. 예식 규모가 작으면 포지션을 줄일 수 있고, 필요하면 한 명이 여러 포지션을 맡을 수 있다.

## 팀원 프로필 규칙

- 관리자와 팀장은 팀원의 성별, 가능한 포지션, 팀장 가능 여부를 수정할 수 있다.
- 팀원은 예식 단위가 아니라 근무일 단위로 신청한다.
- 배정 화면은 가능한 포지션과 성별 조건으로 팀원을 필터링한다.
- 기본 화면은 신청자 목록이며, 필요 시 전체 팀원 보기로 직접 배정할 수 있다.

## 상태값

- 근무일 상태: `모집 중`, `배정 중`, `확정`, `완료`
- 팀원 신청 상태: `미신청`, `신청함`
- 팀원 확인 상태: `미확인`, `확인됨`
- 모집 중 상태에서는 예식 정보와 예상 출퇴근 시간을 공개하지 않는다.
- 확정 후에는 출퇴근 시간, 식 개수, 예식 시간 목록, 배정 포지션을 공개한다.

## 알림 정책

- 확정 배정 시 앱 내부 알림을 생성한다.
- 확정 배정 시 PWA 푸시 알림을 발송한다.
- 팀원이 스케줄 상세를 확인하면 미확인 상태를 확인됨으로 바꾼다.
- PWA 푸시 권한 거부 또는 브라우저 미지원 상황은 화면 내 미확인 근무 표시로 보완한다.

## GitHub Project 운영 기준

제품 로드맵은 하나의 GitHub Project로 관리한다. MVP별로 프로젝트를 나누지 않고 `MVP` 필드로 구분한다.

권장 필드는 다음과 같다.

- `MVP`: `MVP 1`, `MVP 1.1`, `MVP 2`, `Backlog`
- `Type`: `Product`, `Bug`, `Ops`, `Harness`, `Idea`
- `Source`: `PRD`, `Ad-hoc`, `Review`, `User Feedback`
- `Priority`: `P0`, `P1`, `P2`, `P3`
- `Status`: `Inbox`, `Ready`, `In Progress`, `Review`, `Done`, `Blocked`

작업 중 떠오른 아이디어는 즉시 GitHub Issue로 만들고 기본값을 `MVP=Backlog`, `Type=Idea`, `Source=Ad-hoc`, `Status=Inbox`로 둔다. 나중에 triage하면서 MVP 작업으로 승격하거나 보류한다.

## 열린 질문

- `축가` 포지션의 성별 조건을 확정해야 한다.
- 출근/퇴근 시간 자동 계산 규칙을 정해야 한다.
- PWA 푸시 알림 제공 환경과 실패 처리 기준을 구체화해야 한다.
- 초대 링크 만료 시간과 재초대 정책을 정해야 한다.
